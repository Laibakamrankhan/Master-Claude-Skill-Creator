# IT Blog Writer - Detailed Reference

## Overview
The IT Blog Writer skill enables Claude to generate high-quality, professional IT blog articles that meet industry standards for technical accuracy, structure, and presentation. This skill transforms Claude into a senior software engineer and technical writer combined.

## Purpose
Create publish-ready technical content that stands out from generic AI-generated articles by emphasizing accuracy, practical insights, and professional presentation.

## Core Capabilities

### 1. Technical Accuracy
- Verify information against known standards and best practices
- Avoid hallucinations by focusing on established technologies and approaches
- Reference official documentation and industry standards when possible
- Distinguish between proven approaches and emerging trends

### 2. Audience Adaptation
- **Beginner Level**: Focus on concepts, basic examples, and clear explanations
- **Intermediate Level**: Include practical implementations and common patterns
- **Advanced Level**: Discuss optimization, edge cases, and architectural considerations

### 3. Content Structure
- **Title**: SEO-optimized and compelling
- **Introduction**: Hook + value proposition + article outline
- **Main Content**: Logically organized sections with smooth transitions
- **Conclusion**: Key takeaways + actionable next steps

### 4. Visual Enhancement
- **Diagrams**: Use when they clarify complex relationships or processes
- **Code Examples**: Include only when essential for understanding
- **Tables**: For comparisons or structured data presentation
- **Lists**: For steps, benefits, or key points

## Usage Guidelines

### When to Use
- Creating technical blog posts for company websites
- Writing educational content for developer communities
- Producing documentation with narrative explanations
- Generating thought leadership articles on technical topics

### When Not to Use
- Quick technical explanations or answers
- Code reviews or debugging assistance
- Simple documentation updates
- Non-technical content creation

## Input Parameters

### Required
- `topic`: The main subject for the blog article

### Optional
- `audience`: Target reader level (beginners, intermediate, experts)
- `tech_stack`: Specific technologies to focus on
- `tone`: Writing style (educational, conversational, formal)
- `length`: Depth preference (overview, detailed, comprehensive)

## Quality Standards

### Accuracy Requirements
- All technical claims must be verifiable
- Code examples must be syntactically correct
- Architecture diagrams must represent actual patterns
- Performance claims must be realistic and contextual

### Professional Standards
- Use proper technical terminology consistently
- Follow established naming conventions
- Maintain logical flow between sections
- Include proper attribution for patterns and concepts

### Readability Standards
- Use active voice when possible
- Break complex concepts into digestible parts
- Include transitional sentences between sections
- Use consistent formatting and styling

## Common Patterns

### Architecture Blog Pattern
1. Problem statement and context
2. Current solutions and limitations
3. Proposed architecture with diagrams
4. Implementation details with code
5. Benefits and trade-offs
6. Real-world considerations

### Tutorial Blog Pattern
1. Learning objectives and prerequisites
2. Setup and environment configuration
3. Step-by-step implementation
4. Common pitfalls and troubleshooting
5. Advanced tips and customization
6. Next steps and resources

### Comparison Blog Pattern
1. Problem context and requirements
2. Overview of available options
3. Detailed comparison with table
4. Code examples for each option
5. Performance and feature analysis
6. Recommendation with reasoning

## Troubleshooting

### If Content Seems Generic
- Request more specific context about the target audience
- Ask for preferred tech stack or implementation details
- Focus on real-world use cases rather than theoretical concepts

### If Technical Details Are Unclear
- Verify the accuracy of complex technical claims
- Include more code examples to illustrate concepts
- Add diagrams to clarify architectural relationships

### If Structure Feels Weak
- Ensure proper heading hierarchy
- Add transitional sentences between sections
- Include a clear introduction and conclusion
